# 개요


**TPC-C (Transaction Processing Performance Council Benchmark C)**는 트랜잭션 처리 시스템의 성능을 측정하기 위한 산업 표준 벤치마크입니다.

특히 **온라인 트랜잭션 처리 시스템 (OLTP, Online Transaction Processing)**의 성능을 평가하는 데 자주 사용됩니다.

- 목적: 실제 비즈니스 환경을 시뮬레이션해서 데이터베이스 시스템의 처리 능력을 평가함
- 도메인: 도매 회사의 주문 및 배송 시스템을 모델링
- 주요 측정 지표: tpmC (Transactions Per Minute-C, 분당 처리 가능한 "New Order" 트랜잭션 수)

## 1. 시뮬레이션 환경

통 회사를 기반으로 설계되었으며, 이 회사는 다음과 같은 구성요소를 가집니다:

- 고객(Customer)
- 창고(Warehouse)
- 지점(District)
- 주문(Order)
- 배송(Delivery)

## 2. 트랜잭션 종류

TPC-C는 다음 **5가지 트랜잭션 유형**을 정의합니다. 이들 중 "New-Order" 트랜잭션이 성능 측정의 중심입니다.

| 트랜잭션 유형	| 설명 |
| --- | --- |
| New-Order |	고객의 새로운 주문을 생성 |
| Payment	| 고객이 지점이나 창고에 지불 |
| Order-Status	| 고객이 주문 상태 조회 |
| Delivery |	배송 요청 처리 (일괄 배송) |
| Stock-Level |	특정 상품의 재고 수준 점검 |

> 💡 실제 비즈니스처럼 다양한 읽기/쓰기 작업을 포함하고 **있어 읽기/쓰기 비율, 데이터 무결성 유지, 동시성 처리** 등의 요소를 잘 반영합니다.

## 3. 데이터 모델 (스키마)

TPC-C는 9개의 테이블로 구성된 데이터베이스 스키마를 정의합니다.
- Warehouse, District, Customer, Order, OrderLine, Stock, Item, 등

## 4. 성능 측정 방식

- 측정 대상: "New-Order" 트랜잭션을 중심으로 전체 시스템 처리량을 측정
- 단위: tpmC (분당 New-Order 처리 수)
- 요건: 신뢰성 있는 결과를 위해 다른 트랜잭션들도 일정 비율로 수행되어야 함
- 지연 시간: 각 트랜잭션의 응답 시간도 측정됨

## 5. 활용 사례

- DBMS 성능 비교: Oracle, MySQL, PostgreSQL, SQL Server 등
- 하드웨어 성능 테스트: 서버, 스토리지, CPU 성능 벤치마킹
- 클라우드 비교: AWS vs Azure vs GCP 같은 클라우드 DB 환경에서 성능 비교

## 6. 단점 및 비판

- 실제 산업 환경과 완전히 일치하지는 않음
- 결과를 공식적으로 발표하려면 TPC의 승인이 필요

# TPC-C의 사용법과 부하 조절 방법

TPC-C는 공식 사양이 복잡하긴 하지만, 대부분은 벤치마크 도구나 오픈소스 구현체를 통해 쉽게 실행할 수 있습니다.

아래에 실제 사용하는 방법 + 부하 조절 방법을 예제로 정리하겠습니다.

가장 많이 쓰이는 오픈소스 구현체 중 하나는 OLTPBench와 tpcc-mysql입니다.

여기서는 대표적으로 tpcc-mysql 기준으로 설명할게요.

PostgreSQL 용 tpcc-postgres, 다른 도구도 비슷한 방식이에요.

## 1. 준비사항

- 데이터베이스 서버 (예: MySQL, MariaDB, PostgreSQL 등)
- tpcc-mysql 소스코드(GitHub 링크: https://github.com/Percona-Lab/tpcc-mysql)

## 2. 설치 및 빌드

```bash
git clone https://github.com/Percona-Lab/tpcc-mysql.git
cd tpcc-mysql/src
make
```

## 3. 데이터베이스에 테이블 생성

```bash
mysql -u root -p
CREATE DATABASE tpcc;
```

테이블 및 스키마 생성:

```bash
./tpcc_load localhost tpcc root [비밀번호] 1
```

## 4. 벤치마크 실행

```bash
./tpcc_start -h localhost -d tpcc -u root -p [비밀번호] -w 1 -c 10 -r 5 -l 300
```
> -w 은 창고 수(Warehouse). 기본 단위로 중요함!

4. 벤치마크 실행

```bash
./tpcc_start -h localhost -d tpcc -u root -p [비밀번호] -w 1 -c 10 -r 5 -l 300
```

### - 실행 옵션 설명

|  옵션	| 설명 |
|  ---	| --- |
| -h	| DB 호스트 |
| -d	| DB 이름 |
|-u / -p	| 사용자 / 비밀번호 |
| -w	| 창고 수 (부하량에 직접 영향) |
| -c	| 동시 접속 유저 수 (클라이언트 수) |
| -r	| 테스트 전 워밍업 시간 (초) |
| -l	| 테스트 실행 시간 (초) |

## 5. 부하 조절 방법

TPC-C에서 부하(load)를 조절하는 주요 요소는 다음과 같습니다.

1. 동시 사용자 수 (-c)
   - 단일 서버의 처리량을 테스트하려면 10~50명 정도 시작
   - 병렬 부하를 높일수록 CPU/IO 부하 증가

2. 창고 수 (-w)
   - 창고 수 = 데이터 크기 + 병렬성 증가
   - 창고 하나당 약 100MB~200MB
   - 다중 쓰레드 간 충돌 방지를 위해 창고 수 ≥ 쓰레드 수 권장

1. 테스트 시간 (-l)
   - 짧은 테스트로 빠르게 성능을 파악하거나 긴 시간으로 안정성과 병목을 분석 가능

2. 쿼리 지연 추가 (sleep) (옵션 구현에 따라 가능)
   - 고의적으로 지연을 넣어 유휴 구간 시뮬레이션 가능

# TPC-C 결과 지표 해석 방법

일반적으로 TPC-C 테스트 도구는 다음과 같은 **핵심 성능 지표 (KPI)**를 출력합니다:


| 지표 |	설명 | 분석 포인트 |
| --- |	--- | --- |
| tpmC	| 분당 New-Order 트랜잭션 수 (Throughput) |	주요 성능 지표, 높을수록 좋음 |
| Latency |	각 트랜잭션의 평균 응답 시간 |	낮을수록 좋음, 지연 요인 파악 |
| StdDev |	응답 시간의 표준편차 |	안정성 (지연 시간의 흔들림) 측정 |
| Abort Rate |	트랜잭션 실패/충돌 비율	| 동시성 제어 문제, DB Lock 확인 |
| CPU / IO | 사용률	서버 자원 사용량 |	병목 지점 분석에 중요 |

## 1. tpmC (Throughput)

- TPC-C는 New-Order 트랜잭션만 throughput 계산에 포함
- 단위: Transactions Per Minute
- 성능 비교시 가장 핵심이 되는 지표
- 스케일 테스트에서는 `창고 수 / 유저 수`를 늘려도 TPM이 선형적으로 증가하는지를 본다.

> Tip: tpmC가 높지만 지연시간이 크다면 병목이 숨어 있을 수 있음!

## 2. Latency (응답 시간)

- 각 트랜잭션 유형별 응답 시간
   - New-Order, Payment, Order-Status, Delivery, Stock-Level
- 평균 응답 시간은 낮을수록 좋고, 표준편차(StdDev)가 작을수록 안정적인 시스템

> `New-Order 평균 응답 시간 > 100ms` 이상이면 병목이나 락 충돌을 의심해봐야 봐야 합니다.


## 3. Abort / Rollback Rate

- 트랜잭션 중 충돌로 인해 실패하거나 롤백된 비율
- Deadlock, Lock Timeout, Unique Constraint 등이 원인
- 0~2%는 허용되지만, 그 이상이면 DB 튜닝 필요

>  예: 동시 쓰레드 수 ↑ 일 때 abort가 ↑ → 인덱스 부족, 락 경합 가능성

## 4. 자원 사용률 분석 (CPU, Memory, IO)

이건 TPC-C 도구가 직접 출력하진 않지만, `htop`, `iostat`, `vmstat`, `perf` 같은 시스템 도구를 같이 써야 정확한 분석이 됩니다.

- CPU 사용률 100% → DB의 연산 병목
- IO Wait ↑ → 디스크 병목 (e.g. random read/write가 많음)
- Memory 부족 → 스왑 발생 → 응답 시간 ↑
