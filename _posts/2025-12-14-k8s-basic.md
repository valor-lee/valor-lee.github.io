# 개요

이 글은 Mac 환경에서 kind를 이용해 Kubernetes 학습 환경을 빠르게 구성하고, kubectl ↔ kube-apiserver 통신 구조, 그리고 Pod의 조회 및 생성 기본 흐름을 이해하는 것을 목표로 한다.

# 왜 kind인가?

`Kubernetes In Docker`

kind = Kubernetes IN Docker

kind는 Docker 컨테이너 안에 Kubernetes 노드를 띄우는 방식이다. k8s-study-control-plane 컨테이너 내부에는 다음과 같은 컨트롤 플레인 컴포넌트가 **static Pod 형태**로 자동 배포된다.
- kube-apiserver
- kebu-controller-manager
- kube-scheduler
- etcd
- kubelet
- containerd

## kind의 장점

Mac에서 가장 빠르고 가벼움

멀티 노드 구성 가능 (master/worker 생성 가능)

공식 문서 예제 90% 이상을 실행 가능하다고 함

kubeadm보다 가볍지만 구조는 동일


👉 학습용 Kubernetes로 최적

# 3. kubectl과 클러스터 연결 구조

## kubeconfig 구성 요소

~/.kube/config에는 다음 정보가 저장된다.
- Cluster: API Server 주소
- User: 인증 정보 (client cert)
- Context: Cluster + User 조합

```bash
kubectl config get-contexts

CURRENT   NAME             CLUSTER          AUTHINFO         NAMESPACE
*         kind-k8s-study   kind-k8s-study   kind-k8s-study
```

## kubectl 인증 방식(kind 기준)

Client certificate 방식 인증을 사용함

kind가 CA / client cert / key를 전부 생성

kube-apiserver는 그 CA를 신뢰

kube-apiserver는 해당 CA를 신뢰

kubectl은 다음을 증명하며 접근

> “나는 이 CA로 서명된 유효한 클라이언트다”

## 실제 통신 흐름

```
kubectl
  ↓ HTTPS (cert 인증)
kube-apiserver (Docker 컨테이너 안)
  ↓
etcd / scheduler / controller
```

kubectl은:
- Pod 생성
- 상태 조회
- 로그 조회

모든 걸 API Server에 요청할 뿐이고
실제 작업은 컨트롤 플레인 내부 컴포넌트들이 수행함.

# 4. 

## Pod 상태 조회

`kubectl get pods` 를 수행하면 

1. kubectl -> API Server
   - kubectl이 kubeconfig를 읽고 현재 context의 API Server 주소 확인
   - Client Certificate로 인증하고 REST API 호출 (`GET /api/v1/pods`)

2. Kube-apiserver processing
   - Authentication, Authorization, Admission 진행 후 etcd에 저장된 상태 조회

3. etcd에서 상태 조회 
   - /registry/pods/.... 경로에 저장되어 있음

4. 결과반환

>  👉 `kubectl get`은 실행 상태를 보는 것이 아니라, etcd에 기록된 "desired + current state"를 본다.

## Pod 생성

`kubectl run nginx --image=nginx` 를 수행하면,

1. kubectl -> API server
   - `POST /api/v1/namespaces/default/pods`
2. API Server Processing
   - 요청 검증
   - etcd에 "이 Pod를 만들고 싶다"라는 기록 저장
   - 아직까지 Pod는 실행되지 않는 상태의 선언 객체
3. Scheduler Processing
   - etcd에 기록된 pod 중 node에 배치되지 않는 pod가 있는지 watch
   - 적절한 Node를 선택하고 Pod Spec에 nodeName 설정
4. kubelet Processing
   - 해당 Node의 kubelet이 내 노드에 할당된 pod를 발견
   - container runtime에 container 생성 요청

> 👉 Kubernates는 "명령형"이 아닌, "선언형"이다. cmd를 수행하면 k8s는 그 상태를 맞추려 지속 노력한다.
>
> kubectl로 API Server에 요청하고 API server는 etcd에 상태를 저장하고, Controller/Scheduler/kubelet 이 그 상태를 현실로 만든다.

```

kubectl run test --image=nginx

kubectl get pods

test   0/1     ContainerCreating   0          6s

NAME   READY   STATUS    RESTARTS   AGE

```

## Pending Pod

Pod의 생명주기

`Created -> Pending -> Running -> Succeeded/Failed`

Pending은 "아직 실행되지 않았다"는 결과를 의미한다.

### Pending 분류

**Case 1 : Scheduler 단계에서 막힘**

Node를 아직 못 고른 상태

특징
- spec.nodeName가 비워져있음
- Scheduler가 배치 실패중

흔한 원인 
- 모든 Node의 리소스 부족(CPU/Memory)
- taint 있는데 Toleration없음
- affinity 조건 불일치
- Node가 Ready가 아님

이벤트 예시

```bash
kubectl run big \
  --image=nginx \
  --restart=Never \
  --overrides='
{
  "spec": {
    "containers": [{
      "name": "big",
      "image": "nginx",
      "resources": {
        "requests": {
          "cpu": "100"
        }
      }
    }]
  }
}'

kubectl get pods
NAME   READY   STATUS    RESTARTS   AGE
big    0/1     Pending   0          18s

kubectl describe pod big
...
Node:             <none>
...
Events:
  Type     Reason            Age   From               Message
  ----     ------            ----  ----               -------
  Warning  FailedScheduling  26s   default-scheduler  0/1 nodes are available: 1 Insufficient cpu. no new claims to deallocate, preemption: 0/1 nodes are available: 1 Preemption is not helpful for scheduling.

```
- CPU:100이 가능한 Node가 없어서 배치 실패중


- events에 `FailedScheduling`로그 확인됨

**Case 2 : kubelet단계에서 막힘**

Node에 배치되어있지만 실행이 안됨

특징
- `spec.nodeName`이 있음
- 컨테이너 생성 실패

흔한 원인
- image pull 실패
- CNI 네트워크 설정 실패
- 볼륨 마운트 실패

이벤트 예시

```bash
kubectl run badimage --image=nginx:doesntexist

kubectl get pods
NAME        READY   STATUS    RESTARTS   AGE
badimage    0/1     Pending   0          18s


kubectl describe pod badimage
...
Node:             k8s-study-control-plane/172.19.0.2
...
Events:
  Type     Reason     Age   From               Message
  ----     ------     ----  ----               -------
  Normal   Scheduled  12s   default-scheduler  Successfully assigned default/badimage to k8s-study-control-plane
  Normal   Pulling    12s   kubelet            Pulling image "nginx:doesntexist"
  Warning  Failed     10s   kubelet            Failed to pull image "nginx:doesntexist": rpc error: code = NotFound desc = failed to pull and unpack image "docker.io/library/nginx:doesntexist": failed to resolve reference "docker.io/library/nginx:doesntexist": docker.io/library/nginx:doesntexist: not found
  Warning  Failed     10s   kubelet            Error: ErrImagePull
  Normal   BackOff    9s    kubelet            Back-off pulling image "nginx:doesntexist"
  Warning  Failed     9s    kubelet            Error: ImagePullBackOff


```
- kubelet에서 image pull 실패로 인한 pending

> Pod가 Pending이라는 것은 Scheduler 단계거나 kubelet 단계에서 문제가 발생한 것이고, 그 원인은 pod Events log에서 확인할 수 있음
